]]>http://maruthisharma.com/updated-mark-three/edc5e19c-6fe2-4912-aa7b-2d393778f01eTue, 28 Jan 2014 17:04:22 GMT<![CDATA[Ruby - "Jekyll" Blogging Platform Installation Tips for Windows]]>About a week ago, I decided to try out the ruby-based blogging platform (also actually a static site generator) named "Jekyll" .

I realised there are a few issues with it if you try to use it on Windows.

Lets just skip the "address-each-issue" part. I'll simply point out 4 important steps to get it Up-and-Running smoothly :P

  1. Install Ruby 1.9.3 instead of Ruby 2.1.0
    If you are like me. You might always have the temptation of downloading the "latest" version of everything. Well, in this case, you might have to calm down a bit.

    Although 1.9.3 is like the older version, it is supported till February 23, 2015 and 2.1.0 seems to have a couple of issues, especially with windows. Not only jekyll, even the mighty famous we framework rails might give you trouble if you try to run it on Ruby 2.1.0 (on windows, for sure).

  2. Install Jekyll 1.4.2 instead of Jekyll 1.4.3
    The reasons are similar. Just stick with 1.4.2 till things get cleared.

  3. Pygments - the syntax highlighter
    This actually works well with jekyll. But, you'll need Python to run it (obviously to even install it). So, incase you decide not to have it included, there is one extra thing you have to do. Go to the _config.yml file in your jekyll root and set the pygments variable to false.

  4. Install wdm ( Windows Directory Monitor )
    I don't know if everyone will have this problem. But, for some strange reason, in my case, the gem installer didn't automatically install a dependency name wdm. Just to make sure, run 

        gem list

    and if you don't see something like 

        wdm (0.1.0)

    (almost at the end, because the list is usually sorted alphabetically) , then you have to install it. Simply run

        gem install wdm

Done. That's pretty much it.

]]>http://maruthisharma.com/new-project-on-github/434a7616-abe5-4d7b-a52c-21b83b11e9b1Tue, 21 Jan 2014 10:34:54 GMT<![CDATA[How to edit files using notepad++ from WindowsPowerShell]]>So, why PowerShell??.. for using git-nodejs-npm-ish stuff from it. I know Linux’s terminal is better. But, I can’t ignore windows as of now as I use Photoshop, Illustrator and other things on windows. I have nothing against GIMP :P or Inkscape… I just got used to these….

Anyway, I wanted to open a file directly in a custom editor like Notepad++ (which is awesome btw), I knew we could use something like

start notepad++ <file-name>

But (as you can see), its too long. So, I browsed around.I came across some “Scripts” . But I found an easier method to do what I wanted.

All you have to use is an “Alias” ….

So, Open PowerShell, and simply run

Set-Alias

or use its own alias

sal

Under "name" give the shortcut you want to use. I used “npp”.
Now, under path, give the path to Notepad++ Program. You may have to change it according to your directory. Usually its something like

"C:\Program Files (x86)\Notepad++\notepad++.exe"

Now, run

npp <file-name>

And it opens!. Smooth isn’t it?. But you’ll probably curse me when you realise that it doesn’t work anymore after you close your WindowsPowerShell and open it the next time.

In order to save your Aliases permanently, you must store it in your “profile”. Now before we continue, we need to understand some basics about “Profiles”. I’ll simply grab some stuff directly from “Microsoft TechNet’s post” (http://technet.microsoft.com/en-us/library/ee692764.aspx)

What is a Profile?

The Windows PowerShell profile is simply a script file that runs when Windows PowerShell starts up. You can put cmdlets, scripts, functions - any valid Windows PowerShell commands - into this script file. Each time you start Windows PowerShell, this script file will run. That means you can use the profile to set up your Windows PowerShell environment. Typically that would be custom console settings and aliases, but use your imagination and you can come up with other things you’d like to customize in PowerShell before you start working with it.

Finding the profile:

Type this at your PowerShell command prompt:

$profile

That built-in variable will return something like this for Windows XP:

C:\Documents and Settings\keymyer\my documents\WindowsPowerShell\Microsoft.PowerShell_profile.ps1

And this for Windows Vista and Windows Server 2008:

C:\Users\kenmyer\Documents\WindowsPowerShell\Microsoft.PowerShell_profile.ps1

This is the full path to the file that Windows PowerShell will try to run when it starts. Notice we said “try” to run. Here’s an interesting fact: just because you were able to find the profile doesn’t mean it actually exists. $profile is simply a built-in variable that contains the full path to where the profile will be if there is one; the file doesn’t actually have to exist, and, by default. it doesn’t. If you want a profile to run when you start Windows PowerShell, you need to create this file.

Creating a profile:

Before we decide to create a profile, let’s check to see whether we already have one:

Test-Path $profile

If the profile exists this command will return True; if it doesn’t exist, the command will return False. If this command returns False, you need to create the profile.

Creating a profile in Windows XP is really easy. Simply type this at the command prompt:

notepad $profile

This command opens the profile in Notepad. If the profile doesn’t exist, you’ll be prompted to create it. If you choose Yes, the file will be created for you and will be opened in Notepad. Remember, though, we said this is for Windows XP. If you’re running Windows Vista or Windows Server 2008 things get a little more complicated. (But just a little bit, not very much.) If the full path to this file doesn’t exist, trying to open it in Notepad will return an error; you won’t receive any prompts and the file won’t be created for you. The way to create this file in Windows Vista and Windows Server 2008 (an approach which will also work in Windows XP) is to use the New-Item cmdlet:

New-Item -path $profile -type file -force

We pass three parameters to New-Item:

-path $profile We’re passing the full path, stored in the $profile variable, of the item we want to create.

-type file This tells New-Item what type of item we’re creating, in this case a file.

-force This parameter tells New-Item to create the full path and file no matter what.

Now you can open the profile.

notepad $profile

Now, Back to our post

After you’re done with opening your profile, place the following line in it and save.

Set-Alias npp “C:\Program Files (x86)\Notepad++\notepad++.exe”

And like I’ve said before, You may have to change the second argument (the path to .exe file) according to your installation directory.

Done. Now your alias is permanent.

And of course, If you are a fan of Vim or Sublime-Text or Brackets or any other text-editor, you can always add an alias for them. Just make sure the alias isn’t assigned to something. You can run the following command to list out all the existing aliases.

get-alias

That’s pretty much it.
